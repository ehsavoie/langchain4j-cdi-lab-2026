Sous le capot 🔧

1. L'extension scanne pour @RegisterAIService

2. Pour chaque interface trouvée, elle lit les attributs :
chatModelName, tools, chatMemoryProviderName, retrievalAugmentorName

3. Elle crée un bean CDI synthétique :
→ Proxy LangChain4j enrobé dans un bean CDI managé

4. Les composants sont résolus via un SPI (LLMConfig) :
dev.langchain4j.cdi.plugin.<name>.class=...
→ Implémentation MicroProfile Config fournie par langchain4j-cdi-config → Implémentation WildFly fournie par wildfly-ai-feature-pack

5. @Inject Assistant assistant; → ✨ Ça marche !

Détaillons le mécanisme. Au démarrage (ou à la compilation pour Quarkus), l'extension CDI scanne le classpath. Dès qu'elle trouve une interface annotée avec @RegisterAIService, elle crée un bean synthétique. Ce bean est en fait un proxy LangChain4j enrobé dans un wrapper CDI. Quand vous faites @Inject, CDI vous donne ce proxy, et quand vous appelez une méthode, LangChain4j fait le reste : appel LLM, gestion mémoire, RAG, etc. Point important : le cœur de langchain4j-cdi ne dépend PAS de MicroProfile Config. Il utilise un SPI -- LLMConfig -- qui est découvert via ServiceLoader. Ce SPI fournit les clés et valeurs de configuration. MicroProfile Config est juste UNE implémentation possible, fournie par le module langchain4j-cdi-config.
Emmanuel

Toute la configuration est externalisée dans les properties MicroProfile Config. L'interface AI Service


package com.example.demo1;

import dev.langchain4j.service.SystemMessage;
import dev.langchain4j.service.UserMessage;
import dev.langchain4j.cdi.RegisterAIService;

@RegisterAIService(chatModelName = "my-model")
public interface ChatAssistant {

    @SystemMessage("""
        You are a Viking skald performing in the great hall.
        You tell jokes about warriors, longships, axes,
        and the epic chaos of Viking raids. Your humor is sharp,
        self-deprecating, and full of Norse bravado.
        """)
    String chat(@UserMessage String userMessage);
}
SECOURS si la démo ne marche pas. Voici l'interface. C'est un contrat : on définit ce que l'IA fait, pas comment. Le "comment" est géré par CDI et LangChain4j. @SystemMessage définit le comportement de l'IA -- ici un skald Viking qui raconte des blagues. @UserMessage marque le paramètre de l'utilisateur. @RegisterAIService dit à CDI de créer un bean pour cette interface. C'est volontairement léger et humoristique pour montrer qu'on peut créer N'IMPORTE QUEL type d'agent IA avec ce pattern -- un skald Viking, un bot de support technique, ce que vous voulez.

Streaming : réponses en temps réel


@RegisterAIService(streamingChatModelName = "my-streaming-model")
public interface ChatAssistantStreaming {

    @SystemMessage("You are a creative assistant...")
    TokenStream chatStream(@UserMessage String userMessage);
}
    

// Dans l'endpoint JAX-RS avec Server-Sent Events
@GET
@Path("/stream")
@Produces(MediaType.SERVER_SENT_EVENTS)
public void chatStream(@QueryParam("message") String message,
                       @Context SseEventSink sink, @Context Sse sse) {
    TokenStream tokenStream = streamingAssistant.chatStream(message);

    tokenStream
        .onPartialResponse(token -> sink.send(sse.newEvent("token", token)))
        .onCompleteResponse(resp -> { sink.send(sse.newEvent("done", ""));
                                       sink.close(); })
        .onError(error -> { sink.send(sse.newEvent("error", error.getMessage()));
                            sink.close(); })
        .start();
}

Effet ChatGPT : les tokens arrivent au fur et à mesure ! 🎬

Le streaming permet d'afficher les réponses token par token, comme dans ChatGPT. La différence par rapport au mode classique : on utilise streamingChatModelName au lieu de chatModelName, et la méthode retourne un TokenStream au lieu d'un String. Le TokenStream s'intègre parfaitement avec les Server-Sent Events de JAX-RS. Chaque token est envoyé au navigateur dès qu'il arrive du LLM. L'expérience utilisateur est bien meilleure : l'utilisateur voit la réponse se construire en temps réel au lieu d'attendre 10 secondes pour tout recevoir d'un coup. DEMO : montrer stream.html avec une saga Viking qui s'affiche progressivement.

Vision : analyse d'images


@WebServlet("/uploadServlet")
@MultipartConfig
public class ImageAnalyzerServlet extends HttpServlet {

    @Inject
    @Named("vision-model")
    ChatModel visionModel;  // ← Un simple ChatModel !

    @Override
    protected void doPost(HttpServletRequest request,
                         HttpServletResponse response) throws ... {
        Part file = request.getPart("file");

        UserMessage userMessage = UserMessage.from(
            TextContent.from("Describe this image in detail"),
            ImageContent.from(encodeBase64(file.getInputStream()),
                            file.getContentType())
        );

        ChatResponse answer = visionModel.chat(userMessage);
        response.getWriter().write(answer.aiMessage().text());
    }
}

Multimodal = texte + image dans le même message

L'IA multimodale, c'est la capacité d'analyser des images en plus du texte. Avec LangChain4j-CDI, c'est incroyablement simple : on injecte un ChatModel configuré pour utiliser un modèle vision (comme llama3.2-vision ou GPT-4 Vision). Le point important : UserMessage.from() peut prendre plusieurs contents -- TextContent pour le texte, ImageContent pour l'image. L'image doit être encodée en base64. Le modèle vision analyse l'image et répond en texte. Ça marche pour décrire des images, extraire du texte (OCR), identifier des objets, analyser des graphiques... Configuration : dans microprofile-config.properties, on déclare le vision-model avec la classe de modèle vision appropriée. DEMO : montrer image.html, uploader une photo, voir la description générée par l'IA.

Tools : Function Calling


@ApplicationScoped
public class ExpeditionTools {

    @Inject
    ExpeditionService expeditionService;

    @Tool("List available Viking expeditions for a given date")
    public List<Expedition> listExpeditions(
            @P("Date in yyyy-MM-dd format") String date) {
        return expeditionService.getExpeditions(LocalDate.parse(date));
    }

    @Tool("Enroll a warrior in an expedition")
    public Enrollment enrollWarrior(
            @P("Warrior name") String name,
            @P("Desired date") String date) {
        return expeditionService.enroll(name, LocalDate.parse(date));
    }
}

MicroProfile Fault Tolerance


@RegisterAIService(chatModelName = "my-model")
public interface ResilientAssistant {

    @Retry(maxRetries = 3, delay = 1000)
    @Timeout(value = 30, unit = ChronoUnit.SECONDS)
    @Fallback(fallbackMethod = "fallbackChat")
    @CircuitBreaker(requestVolumeThreshold = 5,
                    failureRatio = 0.5)
    String chat(@UserMessage String message);

    default String fallbackChat(String message) {
        return "Service temporarily unavailable. "
             + "Please try again in a moment.";
    }
}

La solution : Guardrails — les garde-fous de vos agents IA

Transition vers les guardrails. C'est une problématique réelle en production : les LLMs peuvent répondre dans la mauvaise langue, glisser du contenu hors sujet, ou ignorer vos contraintes métier. Dans notre démo Viking, le skald est censé chanter en français des chants héroïques nordiques. Sans guardrails, rien ne l'empêche de chanter en anglais, de parler d'elfes ou de nains de Tolkien, ou de produire n'importe quoi. Les guardrails sont là pour ça : valider l'entrée de l'utilisateur AVANT d'appeler le LLM, et valider la réponse du LLM AVANT de la renvoyer au client.
Emmanuel

Filtre les requêtes avant le LLM

✅ Contenu conforme ?

@RegisterAIService(
    chatModelName        = "my-model",
    inputGuardrailNames  = {"french-input",  "fantasy-input"},
    outputGuardrailNames = {"french-output", "fantasy-output"})
public interface SkaldAssistant {

    @SystemMessage("Tu es un skald viking qui compose des chants épiques en FRANÇAIS...")
    String composeSong(@UserMessage String request);
}
Les guardrails s'intègrent directement dans @RegisterAIService. Deux listes : inputGuardrailNames et outputGuardrailNames. Les noms correspondent à des beans CDI annotés @Named. L'ordre dans la liste est important : les guardrails sont exécutés dans l'ordre déclaré. Si un guardrail d'entrée échoue, le LLM n'est jamais appelé. Si un guardrail de sortie échoue, une exception est lancée — le code appelant doit la gérer. C'est élégant : on garde les contraintes métier séparées de la logique de l'assistant. Chaque guardrail a une responsabilité unique.
Emmanuel

Guardrails = Beans CDI nommés


@ApplicationScoped
@Named("french-input")    // ← Nom référencé dans @RegisterAIService
public class FrenchInputGuardrail implements InputGuardrail {

    private LanguageDetector detector;

    @PostConstruct
    void init() {
        detector = new OptimaizeLangDetector().loadModels();
    }

    @Override
    public InputGuardrailResult validate(UserMessage userMessage) {
        String text = userMessage.singleText();
        LanguageResult result;
        synchronized (detector) {       // ← Thread-safe !
            result = detector.detect(text);
        }
        if (result.isReasonablyCertain() && !"fr".equals(result.getLanguage())) {
            return failure("Seul le français est accepté !");
        }
        return success();
    }
}

Apache Tika pour la détection de langue — benefit of the doubt pour les textes courts ou ambigus

Un guardrail c'est un bean CDI — rien de plus. @ApplicationScoped pour partager le détecteur de langue. @Named pour le lier à @RegisterAIService. L'interface InputGuardrail n'a qu'une méthode : validate(). Elle retourne success() ou failure("message"). Point important : le LanguageDetector d'Apache Tika n'est pas thread-safe. Puisque le bean est @ApplicationScoped, plusieurs requêtes peuvent appeler validate() en parallèle. D'où le synchronized. La logique "benefit of the doubt" : si Tika n'est pas raisonnablement certain de la langue (texte trop court, un seul mot comme "Odin"), on laisse passer. On ne bloque que quand on est sûr que ce n'est pas du français.

OutputGuardrail : valider la réponse


@ApplicationScoped
@Named("fantasy-output")
public class NoFantasyRacesOutputGuardrail implements OutputGuardrail {

    private static final Set<String> FORBIDDEN_WORDS =
        Set.of("nain", "nains", "elfe", "elf", "elfes");

    @Override
    public OutputGuardrailResult validate(AiMessage responseFromLLM) {
        String text = responseFromLLM.text().toLowerCase();

        for (String word : FORBIDDEN_WORDS) {
            if (text.contains(word)) {
                return failure(
                    "Le chant du skald ne doit pas évoquer "
                    + "les nains ni les elfes ! C'est une saga viking pure.");
            }
        }
        return success();
    }
}

L'OutputGuardrail reçoit l'AiMessage complète — même pattern que l'InputGuardrail

Le ChatResource la catch et extrait le message d'erreur pour l'afficher à l'utilisateur. Note : on pourrait aussi configurer le nombre de retry pour les guardrails de sortie — LangChain4j peut réessayer automatiquement en informant le LLM de son erreur.

AgentExecutor : recevoir les requêtes A2A


@Produces
public AgentExecutor agentExecutor() {
    return new AgentExecutor() {
        @Override
        public void execute(RequestContext context, EventQueue eventQueue)
                throws JSONRPCError {
            TaskUpdater updater = new TaskUpdater(context, eventQueue);
            if (context.getTask() == null) { updater.submit(); }
            updater.startWork();

            String userMessage = extractTextFromMessage(context.getMessage());
            String response = creativeWriter.generateStory(userMessage);

            updater.addArtifact(List.of(new TextPart(response, null)),
                null, null, null);
            updater.complete();
        }
    };
}

L'AgentExecutor est un producteur CDI — pattern identique à l'AgentCard

SECOURS si la démo live ne marche pas. L'AgentExecutor reçoit les requêtes A2A. Il crée un TaskUpdater pour gérer le cycle de vie de la tâche A2A : submit, startWork, addArtifact, complete. Le message A2A contient des TextParts. On extrait le texte, on appelle le service IA, on renvoie la réponse comme artifact.
